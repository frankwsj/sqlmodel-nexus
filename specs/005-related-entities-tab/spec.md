# 功能规格说明：Voyager ER 图 —— 关联实体侧边栏 Tab

**功能分支**：`005-related-entities-tab`

**创建日期**：2026-06-30

**状态**：草稿

**输入**：用户描述："现在 voyager 中， er diagram tab 中， 如果我点击一个 entity 现在会高亮周围的相关 entity （一层）， 但是我发现尽管这个效果还不错， 但是在 entity 的数量比较大的情况下， 比如 demo/enterprise_voyager 中的例子， 相关联的 entity 还是会分布在很远的地方， 我希望改善这个情况， 解决方案是在 double click 之后出现的侧边栏中， fields/source code 右边添加新的 tab: related entities， 里面发起一个新的查询， 或者前端做某种数据过滤， 效果是 tab 中的内容只展示有关系的 entities。"

## 澄清记录

### Session 2026-06-30

- Q: 侧边栏已打开时，画布上点击其他 entity 应由哪种手势触发侧边栏更新？ → A: 选项 A —— 侧边栏已打开时，**单击**画布上的 entity 即可更新侧边栏（同时保留高亮）；**双击**保留"从关闭状态打开侧边栏"的现有入口，已打开时双击等效于更新。
- Q: 切换实体时侧边栏当前 tab 是否保留？ → A: 选项 A —— 切换实体时**保留**当前激活的 tab，让用户能用同一视角对比多个实体。
- Q: 侧边栏已打开时点击画布空白处会怎样？ → A: 选项 B，并区分手势——纯**点击**（无拖动）关闭侧边栏；**拖拽**（例如平移画布）不关闭。底层沿用现有的 click/drag 区分逻辑。
- 用户追加：砍掉"从列表跳转到关联实体"的故事（原 Story 2）；Related Entities tab 的内容**只读**，不提供任何点击导航或实体选中交互。
- Q: Related Entities tab 的视觉形式？ → A: 选项 A —— **迷你 ER 图（子图）**，渲染"所选实体 + 其直接关联实体 + 它们之间的边"；**复用主图当前的渲染配置**（show module cluster / show methods / edge length 的 Small/Middle/Large），子图本身**不暴露任何独立配置项**；只读。
- Q: 所选实体没有任何关联实体（孤立）时，子图怎么显示？ → A: 选项 B —— 子图渲染 `X` 自身一个孤立节点，并配合一句明确文案（如"该实体没有直接关联实体"）。

## 用户场景与测试 *(必填)*

### 用户故事 1 —— 在迷你子图中浏览所选实体的直接关联实体（优先级：P1）

开发者或数据探索者正在 Voyager 的 ER 图 tab 中查看一个较大的 schema（例如包含数十个实体的 `demo/enterprise_voyager`）。他们双击某个实体，打开现有的侧边栏（当前包含 Fields 和 Source Code 两个 tab）。他们希望看到一张聚焦的**迷你 ER 子图**，图上只有所选实体本身和与它有直接关系的实体（以及它们之间的边）——而不必在主画布上肉眼搜寻散落各处的邻居。子图复用主图当前的渲染设置（show module cluster / show methods / edge length），用户在主图面板调整这些设置时，子图自动跟随更新。

**为什么是这个优先级**：这是本功能要解决的核心痛点。现有的"单层高亮"在视觉上有用，但在规模较大时会失效，因为主图布局会把邻居节点散布到画布各处。一张限定在侧边栏内、聚焦于所选实体的子图，是能独立交付价值、且可独立测试的最小交付单元。

**独立测试**：在 `demo/enterprise_voyager` 上打开 ER 图，双击任意实体 `X`，切换到 "Related Entities" tab，验证子图恰好包含 `X` 及其所有直接关联实体（既无遗漏，也无多余）；再切换主图面板上的 show methods 或 edge length，验证子图跟随变化。

**验收场景**：

1. **给定** ER 图 tab 已加载一个较大的 schema，**当** 用户双击实体 `X` 并选择 "Related Entities" tab，**那么** 子图渲染 `X` 自身、每一个与 `X` 有直接关系的实体、以及它们之间的边；不渲染任何无关实体。
2. **给定** "Related Entities" tab 已为实体 `X` 打开，**当** 子图渲染完成，**那么** `X` 与每个邻居之间的边表达出连接字段与方向，与主图中同类边的渲染规则一致。
3. **给定** 实体 `X` 没有任何关联实体，**当** 用户打开 "Related Entities" tab，**那么** 子图渲染 `X` 自身一个孤立节点，并展示一句明确的"该实体没有直接关联实体"提示文案，让用户能区分这是"无关系"而非"加载中"或"出错"。
4. **给定** "Related Entities" tab 已打开，**当** 当前选中实体变为另一个实体 `Y`（例如双击了 `Y`），**那么** 子图刷新为 `Y` 自身 + `Y` 的直接关联实体。
5. **给定** 侧边栏已经为实体 `X` 打开（无论当前在 Fields / Source Code / Related Entities 哪个 tab），**当** 用户在画布上**单击**另一个实体 `Y`，**那么** 侧边栏立即更新为 `Y` 的内容（Fields、Source Code、Related Entities 均反映 `Y`），同时画布上 `Y` 的邻居高亮仍然保留。
6. **给定** "Related Entities" 子图已为 `X` 渲染，**当** 用户在主面板切换 show module cluster / show methods / edge length（Small / Middle / Large）中任意一项，**那么** 子图按新设置重新渲染；并且子图区域内不出现任何独立的配置控件。

### 用户故事 2 —— 一眼识别关系类型与方向（优先级：P2）

在扫读子图时，用户希望立刻看出每条边是**哪种**关系、朝哪个方向——例如：出向引用（`X` 指向它）、入向引用（它指向 `X`）、共享基类/继承、或双向——从而无需打开每个实体就能推理数据流向。

**为什么是这个优先级**：能显著提升子图的可读性与推理效率，但即使所有边风格一致，子图本身也已交付 Story 1 的价值，因此是 P2。

**独立测试**：为一个同时拥有上游、下游邻居以及至少一个共享基类关系的实体打开 "Related Entities" tab；验证子图中的每条边通过一致的视觉标记（箭头、颜色、线型等）传达其关系类型/方向。

**验收场景**：

1. **给定** 实体 `X` 拥有不同关系类型（入向、出向、共享基类）的邻居，**当** 子图被渲染，**那么** 每条边通过一致且可区分的视觉标记传达其关系类型/方向，且与主图中同类边的视觉语义保持一致。
2. **给定** 两个实体之间通过多于一种关系相连（平行边），**当** 它们出现在子图中，**那么** 每一种独立的关系都被渲染为独立的边，而不是被悄悄合并。

### 边界情况

- 当所选实体**完全孤立**（没有任何关系）时：子图渲染 `X` 自身一个孤立节点 + 一句"该实体没有直接关联实体"的提示文案（见 FR-005）。
- 在子图数据仍在解析/渲染期间（加载状态），tab 应如何表现？
- 如果在 tab 打开期间底层的图数据发生变化（例如 schema 被重新生成），子图会刷新还是变成陈旧数据？
- **自引用**（`X` 引用自身）和**循环**（`A`↔`B`）在子图中如何呈现？
- 对于直接邻居数量非常多的实体（例如 30+），子图如何保持可用（缩放、平移、滚动）？
- 如果关系数据源临时不可用，tab 是否能优雅降级（清晰的错误信息），而不是悄悄显示空白？
- 侧边栏已打开时，用户在画布空白处的**纯点击**（应关闭侧边栏）与**拖拽**（平移视图，不应关闭）如何被正确区分？
- 主图配置（show module cluster / show methods / edge length）变化时，子图是否无遗漏地跟随，且不引入子图自己的配置存储？

## 需求 *(必填)*

### 功能需求

- **FR-001**：Voyager ER 图侧边栏（双击后打开、当前含 "Fields" 和 "Source Code" 两个 tab 的那个面板）必须额外提供一个名为 "Related Entities" 的 tab。
- **FR-002**：当某个选中实体的 "Related Entities" tab 处于激活状态时，**子图必须只渲染**所选实体自身 + 与该实体有直接关系的实体 + 它们之间的边——这里的"直接关系"与现有"点击高亮（一层）"行为所采用的是同一概念；不渲染任何一层以外的实体。
- **FR-003**：子图中连接所选实体与每个关联实体的**边**必须表达出连接字段/边以及相对所选实体的方向，渲染规则与主图中同类边一致。
- **FR-004**：每当选中实体发生变化（即侧边栏被重新打开指向另一个实体，或被重新定向到另一个实体）时，子图必须刷新。
- **FR-005**：当所选实体没有任何关联实体时，子图必须渲染 `X` 自身一个孤立节点，并展示一句明确、可读的提示文案（如"该实体没有直接关联实体"），让用户能区分这是"无关系"而非"加载中"或"出错"。
- **FR-006**：当关联实体数据无法获取时，tab 必须展示一个明确、可读的错误状态，且与空状态相区分。
- **FR-007**：tab 内的子图**必须只读**——不提供任何点击导航、不在子图内选中实体、不改变主画布的当前选中状态。
- **FR-008**：实现可以自由选择通过发起一次专门的查询，或在客户端对已加载的图数据进行过滤来取得关联实体集合；这一选择属于实现层面的考量，但对用户可见的结果必须一致。
- **FR-009**：子图中的每条边必须以一致、可区分的视觉方式传达关系类型/方向（入向、出向、双向、共享基类/继承），并与主图同类边的视觉语义保持一致。
- **FR-010**：同一对实体之间的平行关系（多条独立边）必须各自被渲染为独立的边，而不是被悄悄合并为一条。
- **FR-011**：当侧边栏已经打开（处于可见状态）时，用户在画布上**单击**任意 entity 必须让侧边栏立即更新为该 entity 的内容（覆盖 Fields / Source Code / Related Entities 三个 tab），同时保留现有的"单击 = 高亮一层邻居"行为。**双击**保留其现有职责：在侧边栏关闭时打开它；在侧边栏已打开时双击与单击效果一致（幂等）。此行为修复当前"双击打开侧边栏后，单击其他 entity 侧边栏内容不跟随变化"的问题。
- **FR-012**：当侧边栏因选中实体变化而更新时，当前激活的 tab（Fields / Source Code / Related Entities）**必须保留**，不因实体切换而重置。
- **FR-013**：当侧边栏已打开时，用户在画布**空白处**（非 entity、非 edge）的**纯点击**（mousedown 后无明显位移即 mouseup）必须**关闭侧边栏**，回到无选中状态；若是**拖拽手势**（用于平移/框选画布），侧边栏必须保持不变。click 与 drag 的判定沿用画布现有的逻辑，不引入新的判定机制。
- **FR-014**："Related Entities" tab 的内容形式是一张**迷你 ER 子图**，渲染"所选实体 + 其直接关联实体 + 它们之间的边"，作为主图的高亮一层邻域在侧边栏内的聚焦视图。
- **FR-015**：子图必须**复用主图当前的渲染配置**——show module cluster、show methods、edge length（Small / Middle / Large）——并随主图配置变化自动重新渲染；子图**不暴露任何独立的配置项**，也不维护子图自己的配置存储。
- **FR-016**：子图**只读**——允许视图层面的平移与缩放（不改变选中状态的操作），但不提供实体选中/导航交互，与 FR-007 一致。

### 关键实体 *(涉及数据时填写)*

- **所选实体（Selected Entity）**：用户双击的实体，作为计算"关联实体"的锚点。已存在于当前侧边栏中。
- **关联实体（Related Entity）**：通过一条直接关系与所选实体相连的另一个实体。每个实例携带：标识、关系类型（外键/引用、共享基类/继承等）、方向（入向 / 出向 / 双向），以及连接字段或边。
- **关系（Relationship / Edge）**：所选实体与某个关联实体之间的链接；可能是一对一、一对多或多对多（同一对实体之间可能存在平行边）。
- **关联子图（Related Sub-graph）**：由"所选实体 + 直接关联实体 + 它们之间的边"构成的有界图视图；复用主图渲染配置；只读；随主图配置与所选实体变化而刷新。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**：在 `demo/enterprise_voyager` 量级的 schema 上，用户可以仅凭侧边栏内的子图识别出某所选实体的所有直接关联实体，无需在主画布上平移或缩放。
- **SC-002**：对任意所选实体，"Related Entities" 子图显示其全部直接关联实体（含所选实体自身），且不显示任何无关实体（相对现有"一层高亮集合"达到 100% 精确率与 100% 召回率）。
- **SC-003**：当用户在主面板切换 show module cluster / show methods / edge length（Small / Middle / Large）中任意一项后，子图在本 UI 面板交互语境下被用户感知为即时地重新渲染并反映新配置。
- **SC-004**：对于一个没有任何关系的所选实体，用户能在 5 秒内正确判断该实体是孤立的（即"孤立节点 + 提示文案"毫无歧义）。
- **SC-005**：在切换到一个新选中的实体时，"Related Entities" 子图进入可用渲染状态所花的时间，在本 UI 的面板交互语境下被用户感知为即时。

## 假设

- 本功能中的"直接关联"指的是与 ER 图 tab 中现有"点击高亮（一层）"行为所采用的同一套一层邻域；更深层的（多跳）遍历不在 v1 范围内。
- 视为相关的关系类型包括：引用/外键形式的边，以及共享基类/继承形式的边——与当前图和高亮逻辑已建模的内容保持一致。
- 侧边栏现有的 "Fields" 与 "Source Code" tab 继续正常工作、保持不变；本功能是新增一个同级 tab，而不是替换它们。
- 本功能的范围限定在 Voyager ER 图语境内；其他 tab（如有）不受影响。
- 性能受限于典型 Voyager schema 的规模；预期最大用例大约为 `demo/enterprise_voyager` 的体量，子图预期通过缩放/平移保持可用，而不必引入分页或外部窗口。
- 子图的"只读"指不提供实体选中/导航交互；视图层面的平移与缩放仍然允许，因为它们不改变选中状态、也不构成导航。
- 子图复用主图的渲染管线与配置语义；本功能不为子图引入独立的渲染规则或配置存储，配置的真实来源始终是主面板上的现有控件。
